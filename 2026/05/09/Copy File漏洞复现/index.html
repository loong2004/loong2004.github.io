Copy File漏洞复现

Loong
  2026-05-09 16:13:52 2026-05-09 16:13:52 Created   2026-05-09 16:46:09 2026-05-09 16:46:09 Updated  
  • Linux 
 1.3k Words  4 Mins  

一、 漏洞概览

  • 漏洞代号： Copy Fail（CVE-2026-31431）
  • 漏洞类型： 逻辑漏洞 / 权限提升漏洞 (Local Privilege Escalation)
  • 影响范围： 几乎涵盖过去 9 年内发布的所有主流 Linux 发行版（受影响内核版本可追溯至 2017 年的 4.14 版）。
  • 危害后果： 本地普通攻击者或恶意脚本可通过该漏洞，在无需复杂竞争条件的情况下，稳定提权至 root 权限。

AF_ALG 是 Linux 内核提供的一个接口，允许用户空间程序直接调用内核级别的加密 API。然而，由于该接口在处理零拷贝（Zero-copy）操作时的状态同步缺陷，导致了一个潜伏多年的高危逻辑漏洞——Copy Fail。

二、 ⚠️ 风险提示与环境准备

破坏性极强： 由于 EXP 会直接篡改 /usr/bin/su 等核心文件的页缓存（Page Cache），通常情况下这种修改仅存在于内存中，重启即可恢复。但在某些特定内存压力或内核配置下，被污染的“脏页（Dirty Pages）”有可能会被内核的回写机制（Writeback）同步到磁盘，导致系统 SUID 验证机制被永久破坏。严禁在生产环境测试。

  • 目标环境： 推荐在隔离的测试用 Ubuntu / CentOS / Debian 虚拟机（Kernel 4.9 ～ 6.x）中进行验证。
  • 备份核心文件： 测试前请务必提前备份 /usr/bin/su 等关键 SUID 文件，并做好系统快照。

三、 🔬 漏洞原理

Copy Fail 存在于 Linux 内核的 crypto（加密）子系统（特别是 algif_aead 模块）。与以往依赖特定竞争条件（Race Condition）的复杂提权漏洞（如 UAF 等）不同，Copy Fail 是一个纯粹的逻辑缺陷。

  1. 触发点： 攻击者首先通过 AF_ALG socket 发起一个加密或解密请求。
  2. 零拷贝滥用： 攻击者利用 splice() 系统调用进行零拷贝数据传输。splice() 的设计初衷是为了在文件描述符之间高效移动数据，而无需在用户空间和内核空间之间进行数据拷贝。
  3. 页缓存污染： 在处理该操作时，内核加密子系统未正确校验缓冲区的可写属性。这允许攻击者将受控的恶意指令（如生成 root Shell 的 shellcode）直接注入到任意可读文件（例如 /usr/bin/su）在内存中的页缓存（Page Cache）中。

通过这种方式，攻击者无需修改实际的磁盘文件，即可在内存中改变 SUID 程序的运行逻辑。当其他用户或攻击者自己再次执行 /usr/bin/su 时，系统会直接执行页缓存中被篡改的恶意代码，从而实现 root 提权。

五、 🛡️ 修复与防御方案

鉴于该漏洞利用稳定且门槛极低，建议所有系统管理员尽快采取以下防御和修复措施：

1. 官方补丁（首选方案）

Linux 内核社区及各大发行版（Ubuntu, Debian, RedHat 等）均已发布安全更新。请立即将系统内核升级至安全版本。

Bash

升级后请务必重启系统以加载新内核。

2. 临时缓解措施（Mitigation）

如果生产环境因业务原因暂时无法重启或升级内核，可以尝试以下临时缓解方案：

  • 限制 AF_ALG 访问： 如果系统业务不依赖于用户空间调用内核加密 API（大多数常规应用如 Nginx, MySQL 不依赖此机制），可以通过安全模块（如 AppArmor 或 SELinux）限制普通用户对 AF_ALG socket 的创建权限。
  • 系统调用过滤： 在容器化环境（如 Docker/Kubernetes）中，可通过配置 Seccomp profile，临时禁用或严格限制对 splice() 系统调用在特定条件下的使用（需仔细评估对业务性能的影响）。

3. 入侵检测与监控（蓝队建议）

  • 监控异常 SUID 调用： 密切监控 /usr/bin/su、/usr/bin/passwd 等高危 SUID 程序的异常调用链路。
  • eBPF 行为审计： 利用 eBPF 技术监控内核态中 splice 与 AF_ALG 相关的异常数据流和页缓存写入行为，及时发现漏洞利用尝试。
